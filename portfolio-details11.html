dbt Semantic Layer & Analytics Engineering

Overview

An end-to-end analytics engineering reference stack built on Snowflake — from raw source ingestion through governed, metric-consistent BI consumption. Mirrors the production data platform built at PlanSource connecting Salesforce, NetSuite, and ad platforms into a single source of truth.

What was built

  • Ingestion layer: Fivetran connectors pulling from Salesforce, Google Ads, LinkedIn Ads, and NetSuite into a raw Snowflake schema with no transformations at load time.
  • dbt transformation layer: Staging → intermediate → mart model architecture. Source freshness tests, schema tests (not null, unique, accepted values, referential integrity), and custom data quality tests — 98%+ test coverage across all models.
  • Dimensional modeling: Star schema fact and dimension tables for sales pipeline, marketing attribution, and financial reconciliation — optimized for analytical query patterns.
  • Semantic layer: dbt Semantic Layer exposing governed metric definitions (MRR, churn rate, CAC, LTV) to BI tools — single definition, consistent across every dashboard.
  • CI/CD: GitHub Actions pipeline running dbt test on PR, slim CI for changed models only, and automated deployment to production on merge.
  • Documentation: Auto-generated dbt docs with lineage DAG, column descriptions, and metric ownership — making the warehouse self-documenting.

Why it matters

Analytics breaks down when different dashboards disagree on the same number. A semantic layer solves this at the definition level — not at the BI tool level. This stack demonstrates the engineering discipline (tests, CI, lineage, docs) that separates a maintainable data platform from a collection of SQL scripts.
